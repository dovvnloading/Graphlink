## 1. Module structure and layering

1. **[MED] Flat-import app with package islands.** Root modules import each other by top-level name (`from graphite_window import ChatWindow`); the app only runs with `graphite_app/` as the working directory / sys.path root. There is no installable package, no entry point (`pyproject.toml`/`setup.py` absent) — the "build system" is a Visual Studio `.pyproj`/`.sln`, which is unusual for a Python app and couples contributors to VS conventions.

2. **[MED] Mid-migration split is a standing tax.** Concrete code lives in `graphite_nodes/`, `graphite_canvas/`, `graphite_session/`, `graphite_ui_dialogs/`, `graphite_widgets/`, `graphite_plugins/`, but a parallel layer of top-level compatibility wrappers (`graphite_node.py`, `graphite_canvas_items.py`, `graphite_settings_dialogs.py`, `graphite_core.py`, `graphite_agents.py`, …) must be kept in sync forever. Every symbol now has two import paths and the wrong one is easy to edit (the navigation doc itself has to warn about this).

3. **[MED] Circular-import workarounds instead of fixed layering.** `graphite_window.py:53` does `from graphite_view import ChatView` *inside* `__init__` to dodge a cycle. The window imports every node and plugin class; nodes/plugins reach back up to the window (see §6.2). The dependency graph is bidirectional at almost every layer boundary.

4. **[MED] God modules.** `api_provider.py` (2,164 lines) is simultaneously: provider abstraction for 5 runtimes, filesystem model scanner (Ollama manifests + GGUF), reasoning-text parser, hand-rolled HTTP client for two vendors, image generation, timeout heuristics, and error-message taxonomy. `graphite_settings_dialogs.py` (2,125), `graphite_connections.py` (1,834), `graphite_plugin_gitlink.py` (1,682), `graphite_scene.py` (1,477), `graphite_window_actions.py` (1,432) are all multi-responsibility monoliths.

5. **[MED] `ChatWindow` is a god object assembled from mixins.** `ChatWindow(QMainWindow, WindowActionsMixin, WindowNavigationMixin)` spreads one class across three files (~3,000 lines combined). The mixins silently depend on ~30 attributes created in `ChatWindow.__init__` (`chat_view`, `token_estimator`, `session_manager`, `agent`, `pending_attachments`, …). Neither mixin can be instantiated, reasoned about, or unit-tested independently.

6. **[LOW] Misleading module names.** `graphite_licensing.py` contains no licensing — it is `SettingsManager` ("formerly LicenseManager"). The settings file is named `session.dat` but holds JSON settings, not session state. `graphite_config.py` is not config — it is mutable global runtime state plus theme color helpers (see §2.1). Names actively mislead new readers.

7. **[LOW] Dead/legacy duplicates left in-tree.** `graphite_dialogs.py` (233 lines) duplicates canvas dialog classes but is not the live authority (the navigation doc has to warn people off it). Duplicate class names exist across packages: `NavigationPin` in `graphite_widgets/pins.py` (overlay widget) vs `graphite_canvas/graphite_canvas_navigation_pin.py` (persisted scene item).

8. **[LOW] Eight files in `graphite_widgets/` still carry UTF-8 BOMs** (`__init__.py`, `controls.py`, `overlays.py`, `pins.py`, `scrolling.py`, `text_inputs.py`, `tokens.py`, `tooltips.py`) — tooling that isn't BOM-aware (naive `open()` with `ascii`/`utf-8` strict parsing, some grep/patch flows) trips over them.
    - **[FIXED 2026-07-09]** Stripped the 3-byte BOM prefix from all 8 files (byte-exact, verified the package still imports and exports the same names afterward). Added `tests/test_no_bom_in_widgets.py`, which scans every `.py` file in the package for a leading BOM, so this doesn't quietly come back.

## 2. Global mutable state and configuration

9. **[HIGH] The entire provider runtime is module-level mutable globals.** `api_provider.py:24-48` — `USE_API_MODE`, `API_PROVIDER_TYPE`, `API_CLIENT`, `API_KEY`, `API_BASE_URL`, `LOCAL_PROVIDER_TYPE`, `API_MODELS`, `LLAMA_CPP_SETTINGS` — mutated by `initialize_api()` / `initialize_local_provider()` / `set_task_model()` on the UI thread while `chat()` reads them from many worker threads with no synchronization. Switching modes while any request is in flight can execute a request against a half-swapped provider (e.g., new `API_PROVIDER_TYPE` with old `API_CLIENT`/`API_KEY`). Nothing prevents mode switching mid-request.
    - **[FIXED 2026-07-09] (mode-switch guard)** `on_mode_changed` now refuses to switch while `_main_request_active`, reverts the combo silently, and banners why — closing the one instance of this race a user can trigger from the toolbar with no special timing. Re-selecting the *current* mode while active stays a no-op (no nag) since Qt can fire the signal on programmatic re-sets. Regression tests in `tests/test_mode_switch_guard.py`.
    - **[FIXED 2026-07-09] (per-request snapshot)** Added `_PROVIDER_STATE_LOCK` + an immutable `_ProviderSnapshot`: `chat()` and `generate_image()` capture one consistent view of the provider state at entry (under the lock, which all mutators — `initialize_api`, `initialize_local_provider`, `set_mode`, `set_task_model` — now also take) and route the *entire* request through it: branch selection, client, API key (threaded down through the Gemini upload/post/delete and Anthropic REST helpers), task models, llama.cpp settings (threaded through the whole llama helper chain), and — subtle but real — the error-classification handler at the bottom of `chat()`, which previously re-read the swapped globals and could report an OpenAI quota message for a Gemini request. The llama.cpp model *preload* deliberately happens outside the lock so a multi-GB load never blocks other requests' snapshots. Module globals remain authoritative (so all existing tests/monkeypatching keep working) — the snapshot is a per-request view, not a new state store. Regression tests in `tests/test_provider_state_snapshot.py` simulate the race directly: a stubbed network call flips the globals mid-request and the assertions prove routing, key selection, and error classification all stick with the provider the request started as. **Residual (accepted):** a settings save *between* two independent requests still changes provider for the later request — that's the intended behavior, not a race; and background plugin requests aren't blocked by the UI guard (they get snapshot consistency instead).

10. **[MED] `graphite_config.py` mixes constants with mutable global state.** `CURRENT_THEME`, `CURRENT_MODEL`, and the `OLLAMA_MODELS` dict are mutated at runtime (`set_current_model`, `sync_ollama_task_models`, `apply_theme`). Task→model resolution happens through a global dict that must be manually re-synced from `SettingsManager` after every settings save; the docstring on `sync_ollama_task_models` (graphite_config.py:157-172) is a confession of how fragile this contract is.

11. **[MED] Two competing sources of truth for provider settings.** Persisted state lives in `SettingsManager`; effective runtime state lives in `api_provider` globals and `config.OLLAMA_MODELS`; the UI (`mode_combo`, settings widgets) holds a third copy. They are reconciled by hand at ~6 call sites (`_initialize_mode`, `_initialize_saved_mode_on_startup`, settings-dialog save paths, `sync_ollama_task_models`). Divergence bugs of exactly this kind are already in the git log ("Graphlink-Web always calling qwen3:8b in Ollama mode").

12. **[LOW] Default drift between layers.** `LLAMA_CPP_SETTINGS` defaults `reasoning_mode: "Quick"` (api_provider.py:41) while `SettingsManager` defaults `"Thinking"` (graphite_licensing.py:60). Default models are hardcoded in at least four places (`qwen3:8b` in graphite_config.py:141-148 and graphite_licensing.py:36/120/248; `deepseek-coder:6.7b` in two places; `gemini-3.1-flash-lite-preview` fallback buried inside `chat()` at api_provider.py:1791; `gemini-2.5-flash-image` at api_provider.py:1660).
    - **[FIXED 2026-07-09] (reasoning_mode only)** Traced both "Quick" defaults (the module-level `LLAMA_CPP_SETTINGS` dict and `_normalize_llama_cpp_settings()`'s missing-key fallback) and confirmed neither is actually observable in current usage - `SettingsManager.get_llama_cpp_settings()` always includes a real `reasoning_mode` value before any llama.cpp chat request could read it, so this was a latent inconsistency rather than an active bug. Aligned both to `"Thinking"` anyway, matching `SettingsManager`'s default everywhere else, so the codebase doesn't contradict itself even in code paths nothing currently exercises. Regression tests in `tests/test_llama_cpp_reasoning_mode_default.py`. **Not fixed**: the "default models hardcoded in four places" half of this finding is unchanged - centralizing model-name defaults has real behavioral implications for users (unlike a reasoning-mode toggle) and needs a product decision, not a mechanical fix.

13. **[LOW] Theme change is a duck-typed broadcast.** `apply_theme()` walks `topLevelWidgets()` calling `on_theme_changed` if present (graphite_config.py:118-120); non-top-level widgets rely on manual propagation chains. There is no theme-change signal; the implicit protocol is invisible and unenforced.

## 3. Security

14. **[HIGH] All secrets stored in plaintext JSON.** `~/.graphlink/session.dat` holds the OpenAI, Anthropic, and Gemini API keys plus the GitHub access token, world-readable-by-default, unencrypted, with no OS keychain integration (graphite_licensing.py:84-90, 459-495). The file is also rewritten wholesale on every trivial toggle, multiplying exposure windows.
    - **[FIXED 2026-07-09]** Added `graphite_secrets.py`: Windows DPAPI (`CryptProtectData`/`CryptUnprotectData` via ctypes — zero new dependencies, user-account-scoped) with values stored as `dpapi:<base64>` in the same JSON fields, so the file shape and the #48 atomic-write machinery are untouched. `SettingsManager` encrypts at set-time, decrypts at get-time, and migrates legacy plaintext secrets on first load (plaintext leaves disk immediately, via the atomic save). Design tradeoffs chosen deliberately: DPAPI over `keyring` because the app is Windows-primary (CONTRIBUTING.md) and it adds no dependency; a `session.dat` copied to another machine/account yields `""` ("not configured") for its secrets rather than garbage — losing silent cross-machine portability of secrets is the point; non-Windows platforms fall back to plaintext, i.e. exactly the pre-fix behavior, so nothing regresses. 15 regression tests in `tests/test_secrets_at_rest.py` (roundtrip, no-plaintext-on-disk, legacy migration, no-rewrite-loop, unicode, graceful degradation). Verified all key consumers go through the getters/setters (settings dialog, window init, gitlink token), so the encrypted form never leaks into a provider client.

15. **[HIGH] Py-Coder executes LLM-generated code with full user privileges and no approval gate.** `PythonREPL` (graphite_agents_pycoder.py:29-91) `exec()`s arbitrary code in a subprocess running as the user. The AI-driven flow (generate → execute → repair) runs model output directly. The Execution Sandbox plugin got an explicit approval dialog (`_handle_code_sandbox_approval_request`, graphite_window_actions.py:1035-1061, with a dedicated test), but the older Py-Coder path never did — an inconsistent trust model for the same threat.
    - **[FIXED 2026-07-09]** `PyCoderExecutionWorker` now has the exact `approval_requested`/`approve()`/`deny()` contract the sandbox worker established: `run()` parks on an event after code generation and before anything reaches the REPL; `stop()` also sets the event so a parked worker can never hang shutdown. The main-thread handler (`_handle_pycoder_approval_request`) mirrors the sandbox dialog, shows the exact code, defaults to **No**, and — going one disclosure further than the sandbox — states that repaired versions of the code may run under the same approval (repair-loop iterations don't re-prompt, matching sandbox behavior). Deliberate boundary, pinned by a test: MANUAL mode stays ungated, because there the user authored the code and clicking Run *is* the approval. 7 regression tests in `tests/test_pycoder_approval_gate.py` (deny/stop/approve paths, single-gate-covers-repairs, prose-only responses never prompt, MANUAL boundary).

16. **[MED] The "Execution Sandbox" isolates packages, not the OS.** The approval dialog says so honestly, but the plugin's name, shield icon, and description ("isolated virtual environment") oversell containment: a venv is not a security boundary; generated code still has full user-account access.
    - **[FIXED 2026-07-09] (description only)** Updated the picker-facing description (in both `PLUGIN_REGISTRY` and `PluginPortal._discover_plugins` — kept in sync, plus a new test asserting they stay in sync) to add the same "isolates installed packages, not the operating system" caveat the approval dialog already states honestly. Display name and shield icon left as-is — those are a more subjective design call than the description text, and out of scope for this pass.

17. **[MED] Web content flows into prompts unfiltered, and the safety validator is broken by its own string protocol.** `WebSearchAgent.validate_content` asks the model to answer `SAFE` or `UNSAFE`, then checks `"SAFE" in decision` (graphite_agents_web.py:212) — **`"SAFE" in "UNSAFE"` is `True`**, so an explicit `UNSAFE` verdict passes validation. The whole gate is a no-op unless the model answers something containing neither token. Fetched pages (arbitrary `href`s from DDG results, any scheme/host, no URL validation) are then concatenated into the summarizer prompt — a straightforward prompt-injection channel presented in the UI as a validated pipeline.
    - **[FIXED 2026-07-09]** `validate_content` now checks `"UNSAFE" in decision` first and returns `False` immediately, before falling through to the `"SAFE" in decision` check. Regression tests added in `tests/test_web_agent_validation.py`. The broader "fetched web content isn't otherwise sanitized before reaching the prompt" concern is unchanged and still open.

18. **[LOW] Update signal is unauthenticated remote content.** `UpdateCheckWorker` fetches a raw markdown file from the GitHub `main` branch (graphite_update.py:8) and surfaces the derived message in a UI banner. No signature/pinning; a compromised repo or MITM (HTTPS mitigates) controls banner text and "update available" state. Low impact since nothing auto-downloads.

19. **[LOW] Attachment/temp-file hygiene.** Large pastes and text drops are written to predictable, shared world-tmp paths (`%TEMP%/graphite_paste_attachments`, graphite_window.py:1096) with `errors="ignore"` (silent data mangling) and only cleaned up if the user clears the attachment in-session — orphans accumulate across crashes.
    - **[FIXED 2026-07-09] (encoding only)** Both write sites (`_stage_large_paste_as_attachment`, `_stage_text_context_attachment`) now use `errors="replace"` instead of `errors="ignore"` — an unencodable character (e.g. an unpaired surrogate from a malformed Windows clipboard/drop payload) becomes a visible marker in the saved attachment instead of silently vanishing with no trace. Regression coverage in `tests/test_temp_attachment_encoding.py`. **Not fixed**: predictable shared temp paths and crash-orphaned files are unchanged — a bigger cleanup-policy question, left open.

## 4. Concurrency and threading

20. **[HIGH] Qt scene objects are read from non-GUI threads.** `ChatWorker.run` (graphite_agents_core.py:172-185) — executing inside `ChatWorkerThread`, and *inside a further nested `threading.Thread`* — walks `current_node.parent_node`, `root_node.scene()`, `scene().system_prompt_connections`, and `prompt_note.content`. QGraphicsScene is not thread-safe; concurrent UI mutation (node deletion, `scene.clear()` on chat load/new chat) can race and crash or read torn state. The same pattern applies to any worker handed a live node.

21. **[HIGH] Single-slot worker attributes drop references to running threads.** `ChatWindow` tracks one thread per attribute (`self.takeaway_thread`, `explainer_thread`, `chart_thread`, `image_gen_thread`, `code_exec_thread`, `pycoder_agent_thread`, `pycoder_exec_thread`, `web_worker_thread`, `conversation_node_thread`). Starting a second web node while the first is running (multiple `WebNode`s exist per scene, each with a Run button) overwrites `self.web_worker_thread` — the old QThread loses its owner reference (GC/`deleteLater` while running crashes Qt) and can no longer be stopped or awaited at shutdown. The sandbox path was already converted to per-node `worker_thread` because of exactly this bug (the comment at graphite_window_actions.py:1063-1068 documents it), but takeaway/explainer/chart/image/web/pycoder still have it.
    - **[FIXED 2026-07-09] (PyCoder only)** `stop_pycoder_node` was the one instance of this pattern that was an actual "wrong node stopped" correctness bug (matching the sandbox bug exactly): it took a specific `pycoder_node` argument but stopped `self.code_exec_thread`/`self.pycoder_exec_thread` regardless of which node was passed. `PyCoderNode` now has its own `worker_thread` attribute (mirroring `CodeSandboxNode`); `execute_pycoder_node`/`stop_pycoder_node` operate on it instead of the shared window attributes, which have been removed. Regression tests in `tests/test_per_node_worker_threads.py`. **Not fixed**: web/takeaway/explainer/chart/image_gen/group_summary still use single-slot attributes — but none of those have a "stop" affordance at all (no wrong-node-stopped risk), so the only remaining gap for them is the shutdown-wait completeness described in #24, not a stop-correctness bug. Also unfixed: `PyCoderAgentWorker` (the post-execution analysis phase chained after `CodeExecutionWorker` in MANUAL mode) has no `.stop()` method at all, so clicking "stop" during that phase is still a no-op — a missing-feature gap, not a wrong-node bug, and not addressed here.

22. **[MED] Cancellation is cosmetic for in-flight network calls.** `_raise_if_cancelled` checks only before/after blocking calls (`ollama.chat`, `urllib.request.urlopen`, `create_chat_completion` — api_provider.py:1735-1777, 1224-1251). "Cancel" leaves the request running to completion (up to 180 s chat / 1,800 s audio-upload timeouts); the watchdog thread in `ChatWorkerThread` (graphite_agents_core.py:63-83) abandons the nested daemon thread rather than stopping it, leaking an active request and its callback context.

23. **[MED] Thread-in-thread watchdog.** `ChatWorkerThread.run` spawns a raw `threading.Thread` inside a QThread purely to implement a timeout, with results passed via closure dicts. Timeouts/cancels return from the QThread while the inner thread keeps running; a late result is silently discarded, a late exception is swallowed. Timeout constants (35 s warn / 420 s fail / 1,800 s audio) are hardcoded (graphite_agents_core.py:104-107).

24. **[MED] Shutdown logic enumerates thread attribute names by hand, including ghosts.** `_iter_shutdown_threads` (graphite_window.py:443-472) lists 20 attribute names — five of which (`reasoning_thread`, `workflow_thread`, `graph_diff_thread`, `quality_gate_thread`, `code_review_thread`) belong to plugins that were deleted. Per-node workers (sandbox/artifact/gitlink/conversation `node.worker_thread`) are *not* in the list, so `closeEvent` never waits for them. Shutdown requests are duck-typed (`cancel`/`stop`/`requestInterruption`) with all exceptions swallowed.
    - **[FIXED 2026-07-09]** Turned out to be 8 ghost names, not 5 (`sandbox_thread`/`artifact_thread`/`gitlink_thread` were also already dead by the time this was written) plus 2 more created by the #21 PyCoder fix (`code_exec_thread`/`pycoder_exec_thread`) — all 10 removed from `_iter_shutdown_threads` and from `ChatWindow.__init__`. `_iter_shutdown_threads` now also walks `scene.code_sandbox_nodes`/`artifact_nodes`/`pycoder_nodes`/`gitlink_nodes` and yields each node's own `worker_thread` if running, so `closeEvent` actually waits for (or reports) them instead of silently abandoning them — e.g. a running sandbox subprocess is no longer abandoned mid-install on app close. Refactored the yielded tuple from `(attr_name, label, worker)` to `(label, worker, clear_ref)` (a callable) since node-scoped workers don't live at a `self.attr_name` path. Regression tests in `tests/test_shutdown_thread_enumeration.py`. **Not fixed**: web/takeaway/explainer/chart/image_gen/group_summary/conversation still only check their single window-level attribute, so concurrent instances of those still only get the most-recently-started one waited on at shutdown; duck-typed shutdown requests with swallowed exceptions is unchanged.

25. **[MED] `stop()` on most workers is a flag, not a stop.** `ExplainerWorkerThread`, `KeyTakeawayWorkerThread`, `GroupSummaryWorkerThread`, `WebWorkerThread` set `_is_running = False`, which only suppresses signal emission; the blocking LLM/HTTP call underneath continues. `PythonREPL.execute` has an unbounded `readline()` loop — code that prints nothing and never exits hangs the worker forever (graphite_agents_pycoder.py:79-83); the only escape kills the whole REPL and its accumulated state.

26. **[LOW] `stop_pycoder_node` stops the shared thread slots unconditionally** (graphite_window_actions.py:920-924) — with multiple PyCoder nodes, stopping one can kill another's execution (same class of bug the sandbox comment describes).
    - **[FIXED 2026-07-09]** Duplicate of #21's PyCoder case (this list was compiled in one read-through pass and the same bug got filed twice, once in the concurrency section and once in the security/threading roundup). Already fixed — see the note under #21.

## 5. Provider layer (`api_provider.py`)

27. **[MED] Error classification by substring-matching exception text.** The `chat()` catch-all (api_provider.py:1882-1962) branches on `"429" in error_str`, `"timed out"`, `"connection refused"`, `"capabilit"`, etc. across five different providers' error formats. This is brittle (messages change between SDK versions, locales) and already needs vendor-specific special cases stacked on top.

28. **[MED] No streaming anywhere.** Every provider path is a blocking full-completion call. For a chat canvas app this means: no incremental rendering, cancel can't take effect mid-generation, and long generations look like stalls (hence the watchdog + "still running" banner machinery, which exists to paper over the absence of streaming).

29. **[MED] Hand-rolled HTTP clients for Anthropic and Gemini.** Raw `urllib.request` with manual header construction, manual error-payload parsing, a hand-implemented resumable upload protocol with a `requests`/`urllib` dual path (api_provider.py:1426-1507), pinned `anthropic-version: 2023-06-01`, and a hardcoded static Gemini model list (`GEMINI_MODELS_STATIC`) that will silently age. The official SDKs (`anthropic` is even in requirements.txt; `google-generativeai` too) are unused or half-used — `initialize_api` tries the Anthropic SDK, falls back to REST, so two divergent code paths must both be maintained.

30. **[MED] `google-generativeai` is a dead dependency.** It is listed in requirements.txt but never imported anywhere; Gemini is spoken to via raw REST. ## 7. Persistence and data model

47. **[HIGH] Graph references are list indices, and load-time skips corrupt them.** Connections and parent/child links serialize as positions in `all_nodes_list` (`all_nodes_list.index(node)`, serializers.py:45-99, 111). On load, unknown/failed nodes are *skipped* (deserialize_node returns None) while `frame_source_map` re-derives chart indices as `len(frame_source_map) + i` and `all_items_map` re-enumerates whatever survived (deserializers.py:537-549). Any skipped node shifts every later index: frames/containers silently adopt the wrong members and connections attach to the wrong nodes — not a crash, a silent graph corruption. There are no stable node IDs (UUIDs).
    - **[FIXED 2026-07-09]** Two-part fix. **(1) Skip-safe arithmetic — protects EXISTING saves:** `restore_chat`'s chart/note/frame slot offsets now come from the original payload counts (`len(node_payloads)` etc.) instead of survivor counts, matching the save-side item space exactly; and `chat_nodes_map` (system-prompt/group-summary links) is keyed by chat-payload position rather than loaded-`scene.nodes` position. With zero skips the new keys are identical to the old ones, so unaffected saves load byte-identically. **(2) Stable IDs — hardens FUTURE saves:** every node payload now carries an `id` (uuid4, stored on the node and restored on load so it survives load/save cycles), and connections/children dual-write `*_id` fields alongside the legacy positional fields; the deserializer prefers IDs with per-position index fallback for pre-ID saves. The decisive regression test (`tests/test_persistence_node_ids.py`, 8 tests) reproduces the exact old corruption — a container referencing the first note silently adopting the *second* note after a plugin-node skip — in both ID-bearing and stripped-to-legacy payload forms, plus ID round-trip stability and IDs-beat-stale-indices. **Refined during the fix:** basic connections and parent refs were *already* skip-safe (`all_nodes_map` is keyed by original payload index — the original finding overstated that part); the real corruption lived in `frame_source_map`/`all_items_map`/`chat_nodes_map`. **New sub-finding turned up while tracing:** a plugin node whose *parent* appears later in `NODE_LIST_NAMES` order (e.g. a Web node branched from an HTML node) is silently dropped on load, because parents are resolved during single-pass construction — a forward-reference limitation that predates this change and is not fixed by it (IDs don't help; it needs two-pass construction). Left open.

48. **[HIGH] Settings writes are non-atomic; a crash mid-write nukes all settings.** `SettingsManager._save_state` truncates and rewrites `session.dat` in place (graphite_licensing.py:161-167); `_load_state` responds to any `JSONDecodeError` by **silently recreating defaults** (graphite_licensing.py:113-114) — a power cut during any toggle-save destroys every API key and preference with no backup and no error surfaced. No temp-file+rename, no versioning.
    - **[FIXED 2026-07-09] (atomicity + backup, not versioning)** `_save_state` now writes to a temp file next to `session.dat` (same directory, so the rename is guaranteed same-volume), `fsync`s it, then `os.replace`s it into place - `os.replace` is atomic on both Windows and POSIX, so `session.dat` itself is now always either the complete old version or the complete new version, never a partial write. Separately, `_load_state`'s `JSONDecodeError`/`IOError` handler now backs up the unreadable file (to `session.dat.corrupted-<timestamp>`) before resetting to defaults, with a printed warning, instead of silently discarding it. Regression tests in `tests/test_settings_atomic_write.py`, including one that simulates a mid-write failure (`os.fsync` raising) and confirms the original file is byte-for-byte untouched afterward. **Not fixed**: no schema version field (see #49) - a future format change still has no migration story, atomic writes just stop *corruption*, not *incompatibility*.

49. **[MED] No schema version anywhere.** Neither the chat payload nor `session.dat` carries a version field. Migration is ad-hoc: 40 sequential `if key not in state` checks duplicating every default a second time (`_load_state` vs `_create_initial_state`), and the deserializer probes three legacy payload shapes inline (`nodes` / `items` / nested `data.nodes`, deserializers.py:502-518). Every schema change adds another archaeological layer.
    - **[FIXED 2026-07-09] (the version field itself, not a migration framework)** Added `SettingsManager.CURRENT_SCHEMA_VERSION` (backfilled into `session.dat` the same way every other key here already is - in memory immediately, persisted on the next `set_*()` call, matching the existing pattern) and `graphite_session.scene_index.CURRENT_CHAT_SCHEMA_VERSION` (written into every newly-serialized chat payload). Both are just the marker - no code reads or branches on either value yet, so this doesn't touch the 40-key ad-hoc backfill list or the deserializer's three-shape legacy probing, which are unchanged. Regression tests in `tests/test_schema_version.py`.

50. **[MED] Whole-graph JSON blob per save, images inlined as base64.** Each save re-serializes the entire scene — every node, every connection, and every image re-encoded to base64 inside the `chats.data` TEXT column (content_codec.py, serializers.py:142) — and `save_chat()` fires after *every* response, plugin completion, and cancellation. Large canvases with images mean multi-MB JSON rewrites dozens of times per session, all serialized on the UI thread (only the DB write is backgrounded).

51. **[MED] Conversation history is duplicated per node.** Every conversational node stores its own deep-cloned full `conversation_history` (clone-on-read *and* clone-on-write in `graphite_memory.py`), and the serializer persists each copy. Regression tests in `tests/test_save_chat_atomically.py` don't just check the happy path - they force a `KeyError` partway through the notes/pins writes (a malformed note/pin dict) and confirm the chat row *and* the already-written notes are rolled back too, not left as a partial commit.

Either the feature regressed silently or the code and docs both need pruning.
    - **[FIXED 2026-07-09]** Restored rather than pruned, since `TitleGenerator` was fully implemented, already documented as live behavior, and the missing wiring looked like an accidental regression rather than an intentional removal. `SaveWorkerThread.run()` now calls `title_generator.generate_title(first_message)` for new chats (and for saves whose `current_chat_id` no longer resolves), falling back to the plain first-five-words title only if the generator is missing, raises, or returns nothing usable. 54. **[LOW] Split persistence models.** Nodes/frames/containers/charts/view-state live in the JSON blob; notes and pins live in dedicated SQLite tables; settings live in a JSON file; sandbox venvs live elsewhere on disk. Backup/export/sync of "a chat" requires knowing all of these.

## 8. Prompt/response protocol

56. **[MED] The model-facing protocol is regex-scraped free text.** Responses are mined with regexes for ```fences```, custom `<code_block>` tags, `<think>`/`<thinking>` tags, `--- REASONING ---` markers, and Harmony channel tokens (`_parse_response` graphite_window_actions.py:393-429; `split_reasoning_and_content` api_provider.py:461-507). Multiple code blocks are joined into one CodeNode with a `# --- Next Code Block ---` comment, keeping only the first block's language. History entries embed pseudo-markup (`--- EXECUTED PYTHON CODE ---`, `--- SANDBOX REQUEST ---`) that later stages re-parse. Fragile by construction, and the SAFE/UNSAFE bug (§3.4) is this pattern's proven failure mode.

57. **[MED] Silent intent hijacking by keyword heuristic.** `_looks_like_image_generation_request` (graphite_window_actions.py:159-172) reroutes ordinary chat messages containing verb+noun pairs like "make … a poster" into image generation with no user control and no way to opt out other than rephrasing.

58. **[LOW] Triplicated `clean_text` markdown strippers.** `ExplainerAgent`, `KeyTakeawayAgent`, and `GroupSummaryAgent` each carry a near-identical ~40-line cleaner (graphite_agents_core.py:285-346, 417-477, 561-605) that also destroys legitimate content (`*`, `_`, backticks stripped from prose/code).
    - **[FIXED 2026-07-09] (dedup only)** Extracted to one shared `clean_agent_markdown_response()` helper. Before touching any of the three, captured golden outputs from the *original* implementations across ~9 representative inputs and confirmed the refactored versions are byte-identical. That comparison actually turned up a genuine (narrow) behavioral difference between the three originals: `GroupSummaryAgent` reset its bullet-list tracking state on a section-marker line, the other two didn't — only visible as one extra blank line in the specific case of [bullet, bullet, section-marker, bullet]. Preserved exactly per-agent via a `reset_bullet_state_on_section_header` parameter rather than silently unifying it. Net effect: -48 lines in `graphite_agents_core.py` despite the new shared function. The "destroys legitimate `*`/`_`/backtick content in prose or code" complaint itself is unchanged (that's the stripper's original design, not something this dedup pass touched) — still open if that behavior itself needs to change.

59. **[LOW] Token budgets are folklore constants.** `max_tokens=8000/7000/6500`, `system_prompt_estimate=500/1000/1200/1600`, `reserve_tokens=min(5000, …)` scattered across send/web/sandbox/artifact/gitlink paths with no relation to the active model's actual context window (Llama.cpp `n_ctx` is user-configurable but ignored by trimming).

## Top themes (if prioritizing)

1. **Thread safety**: Qt objects touched from worker threads (§4.1) and provider globals mutated under live requests (§2.1) are the two most likely sources of "random" crashes.
2. **Data integrity**: index-based graph persistence with skip-on-load (§7.1) and non-atomic settings writes (§7.2) can silently destroy user data.
3. **Security posture**: plaintext keys (§3.1) and ungated Py-Coder execution (§3.2) are the gap between what the UI promises and what the code does.
4. **Model/view separation**: widgets-as-data (§6.1) underlies serialization-on-UI-thread, reflection scraping, and the untestability of nearly everything.
5. **Protocol robustness**: stringly-typed LLM protocols (§8.1) already produced a real bug (§3.4); more are latent.
